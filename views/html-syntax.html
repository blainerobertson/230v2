HTML5 Syntax

Definition

Syntax refers to how we write HTML5 markup. This is not the same as semantics, which is choosing the correct elements to use to structure the content of the page.

Best Practice

HTML5 syntax, according to the W3.org, is a jumbled mess that basically indicates that HTML5 can be written any number of ways. The problem is that the code that you write more often than not will end up being viewed and possibly used by someone else. When code is messy, it is also hard to understand and maintain. Therefore, rather than using the W3.org recommendation, we will use "convention" instead. Remember that "convention" means the way most people do it (see Louis Lazaris' article on this topic). This is also referred to as "best practice".

Four Syntax Rules

  1. All elements and attributes are lowercase.
  2. All attribute values are quoted.
  3. All elements are closed.
  4. Elements close in proper order.

An Explanation

Elements and attributes are lowercase

<a href="/homework/plan.html" title="View the planning file">Plan</a>

Notice how the "a", "href" and "title" are all lowercase? The "a" (or anchor) is the element name and is part of both the opening and closing tags. The "href" and "title" are attributes. They describe the anchor element. By writing them all lowercase, it makes writing the code easier and helps avoid mistakes by capitalizing something and not capitalizing something else. It helps your code be consistent.

Attribute values are quoted

<a href="/homework/plan.html" title="View the planning file">Plan</a>

If you look closely at the path used with the "href" attribute and the sentence used in the "title" attribute you will see that both are enclosed in quotes. It doesn't matter if the quotes are double (") or single('). By surrounding the values with quotation marks it makes it clear that these are values of the attributes.

Elements are closed

There are two types of elements in HTML5 - non-empty and void. Depending on which kind is being used, they close (are ended) in different ways.

Non-empty

Most HTML elements are non-empty. This means they have an opening tag and a closing tag with content between the two. Look at this example:

<a href="/homework/plan.html" title="View the planning file">Plan</a>

The word "Plan" is the content placed between the opening and closing tags. The closing tag closes the element.

Void

A void element contains no content and thus does not need a closing tag. It consists only of an opening tag. HTML5 only has 15 of these [area, base, br, col, command, embed, hr, img, input, keygen, link, meta, param, source, track, wbr] and most are seldom used. The "br", "link", "hr", "img" and "input" are the most common. Consider the examples below:

<img src="/images/map.jpg" alt="An image of the planning site map">

<meta charset="UTF-8">

Note how neither has a closing tag and there is no content outside of the element. In XHTML it was common to include a forward slash "/" at the end of the element just prior to the greater than symbol ">". In HTML5, the slash is not required and is discouraged. In this case the greater than symbol ">" closes the element.

Elements close in proper order

When building a web page and the content within the page it is common to be opening and closing elements throughout the file. This rule means that those elements must close in the same order as they open.

The rule of thumb is First in, last out. Look at the following example:

<p><b>Halt!</b> The guard shouted. <q>Who goes there?</q></p>

You notice that there is a lot of markup in this short passage. But, notice that the "p" opens first and closes last. The "b" opens next but closes after the word "halt". The quote opens before the challenge of "Who goes there?" and closes immediately after. This demonstrates that elements have to open and close in their proper order. Any other order would result in errors being generated.

Summary

By following these four simple syntax rules when writing your HTML markup, your code will be easier to write and read, easier to maintain and put you in the company of thousands of developers who follow best practice.
